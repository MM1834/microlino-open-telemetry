# SPR-0004B.3 Phase 2 — Review

## Result

The package implements the browser side of Cognito Authorization Code flow with PKCE and keeps authentication separate from the AWS data provider.

## Architecture review

- `auth-manager.js` owns flow orchestration, callback validation and token exchange.
- `pkce.js` owns cryptographically secure random values and S256 challenge generation.
- `token-store.js` owns persistence and expiry checks.
- `app.js` owns UI and startup gating.
- `aws-backend-provider.js` remains a data provider and receives tokens through dependency injection.

This preserves the Phase 1 boundary: authentication is a dependency of the AWS provider context, not a data provider itself.

## Security review

Implemented:

- Authorization Code flow, not implicit flow.
- PKCE S256 with a high-entropy verifier.
- State validation before token exchange.
- Nonce generation and validation when the ID token contains a nonce claim.
- Issuer and audience checks when corresponding ID-token claims are present.
- Exact redirect URI reuse during authorize and token requests.
- No client secret in browser code.
- No token logging.
- Callback URL cleanup.
- Session-scoped storage instead of persistent local storage.

Important boundary: browser-side JWT decoding is not signature validation. API Gateway remains responsible for cryptographic JWT validation and authorization enforcement.

## Known constraints

1. Cognito/User Pool/App Client/API Gateway infrastructure is not included and must already be configured.
2. Refresh-token use is deliberately excluded. Expired sessions require a new login.
3. A provider-side 401 updates the UI but does not automatically clear the session or redirect.
4. The dashboard currently has no dedicated full-page unauthenticated view; the existing dashboard remains visible without data.
5. Token persistence lasts only for the current browser session.
6. End-to-end HTTP 200 cannot be proven offline; it depends on deployed Cognito and API Gateway settings.

## Recommended next work

- Add refresh-token rotation or silent reauthentication as a separate sprint.
- Add user-to-vehicle authorization and deny-by-default API logic.
- Add automated browser tests for redirect/callback scenarios.
- Consider a dedicated authenticated shell/loading state when the dashboard UI is next refactored.
