# SPR-0004B.3 — Dashboard Authentication (PKCE)

## Status

- Phase 1 architecture and contracts: **Complete**
- Phase 2 Cognito PKCE browser implementation: **Complete**
- Integration against deployed Cognito/API Gateway: **Pending integrator validation**

## Goal

Implement Cognito User Pool Authorization Code flow with PKCE for the static
browser dashboard and inject the resulting access token into the AWS Backend
Provider.

## Delivered in Phase 2

- cryptographically secure PKCE verifier and S256 challenge;
- state and nonce transaction values;
- explicit Hosted UI login redirect;
- callback parsing and state validation;
- authorization-code exchange at the Cognito token endpoint;
- access, ID and refresh-token response storage in `sessionStorage`;
- access-token expiry checks and expired-session cleanup;
- callback URL cleanup;
- explicit login/logout controls;
- authenticated startup gate for `aws-backend`;
- bearer-token injection into Vehicle API requests;
- structured HTTP 401 notification;
- integration, validation and review documentation.

## Non-goals

- Refresh-token rotation or silent reauthentication
- User-to-vehicle permissions
- Roles or vehicle sharing
- Cognito Identity Pool
- Cognito/API Gateway CloudFormation changes
- Browser-side JWT signature validation

## Runtime flow

1. Dashboard loads configuration and creates `AuthManager` for `aws-backend`.
2. `restoreSession()` handles a callback when `code` or `error` is present.
3. A valid, non-expired session allows provider startup.
4. Without a session, the provider stays stopped and the user sees `Anmelden`.
5. `login()` creates verifier, challenge, state and nonce, then redirects to Cognito.
6. Callback state is checked before the authorization code is exchanged.
7. The stored access token is supplied to the AWS provider through
   `getAccessToken()`.
8. `logout()` clears browser state and redirects through the Cognito logout
   endpoint.

## Security boundary

The dashboard decodes selected ID-token claims only for consistency checks. It
does not perform cryptographic JWT validation. API Gateway's JWT authorizer is
the enforcement point before Lambda.

## Validation criteria

- No JavaScript syntax errors.
- Legacy MQTT starts unchanged without authentication UI.
- Unauthenticated AWS startup does not call `/api/vehicles`.
- Authorize redirect uses code flow, S256 PKCE, state and nonce.
- Callback state mismatch is rejected.
- Successful callback stores a session and cleans the URL.
- API requests carry a bearer access token.
- Expired sessions are cleared and require a new login.
- Logout clears local state and invokes Cognito logout.
- Tokens never appear in application logs.

## Rollback

Revert the Phase 2 commit. No AWS infrastructure rollback is required because
this sprint changes only the static dashboard and documentation.
